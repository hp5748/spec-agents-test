# 实现笔记

## 技术决策记录

### 2024-01-11: LLM 选择
**决策**: 使用硅基流动的 DeepSeek-V3.2 替代通义千问

**原因**:
1. OpenAI 兼容接口，降低迁移成本
2. DeepSeek 中文能力强，性价比高
3. 硅基流动提供稳定的 API 服务

**实现**:
```python
self.llm = ChatOpenAI(
    model="deepseek-ai/DeepSeek-V3.2",
    api_key=Config.API_KEY,
    base_url="https://api.siliconflow.cn/v1",
)
```

---

### 2024-01-11: 流式响应实现
**决策**: 使用 Server-Sent Events (SSE) 实现 HTTP 长连接

**原因**:
1. SSE 比 WebSocket 更简单，适合单向数据流
2. 原生 HTTP 协议，无需额外握手
3. 浏览器原生支持 EventSource API
4. FastAPI 原生支持 StreamingResponse

**实现要点**:
1. 后端使用 generator 生成 SSE 格式数据
2. 设置 `media_type="text/event-stream"`
3. 设置 `X-Accel-Buffering: no` 禁用 nginx 缓冲
4. 前端使用 `fetch` + `ReadableStream` 读取

**SSE 格式**:
```
data: {"type": "content", "content": "文本"}\n\n
```

---

### 2024-01-11: 前端架构
**决策**: 使用纯 HTML/CSS/JS，不引入框架

**原因**:
1. 单页面应用，无需复杂状态管理
2. 减少构建依赖，降低复杂度
3. 加载更快，无需打包过程

**实现亮点**:
1. CSS 变量统一管理颜色主题
2. CSS 动画实现视觉效果
3. SSE 原生 API 处理流式数据
4. 无第三方库依赖

## 已知问题

### 1. 工具调用场景流式效果
**问题**: 订单查询等工具调用场景无法实现真正的流式效果

**原因**: 工具返回的是完整结果，不是流式数据

**临时方案**: 模拟流式输出，将结果分块返回

**待优化**: 考虑让工具返回 generator

### 2. 内存存储
**问题**: 对话历史存储在内存中，重启丢失

**原因**: Demo 级别，未引入持久化存储

**待优化**:
1. 使用 Redis 存储会话
2. 或使用文件持久化

### 3. 并发限制
**问题**: 单进程单线程，并发能力有限

**原因**: uvicorn 默认配置

**待优化**:
1. 使用 gunicorn + uvicorn workers
2. 或使用异步优化
